# 채팅 4회 제한 정책 (Chat 4-Message Limit Policy)

이 문서는 **만나기 전 총 4회 메시지 제한** 정책을 서버 강제 기준으로 정의한다.  
구현이 아닌 **정책·설계 문서**이며, 결제/코인 로직과 독립된 채팅 도메인 정책이다.

---

## 1. 정책 정의

| 항목 | 내용 |
|------|------|
| **누가** | 매칭된 한 쌍의 유저(카드 수신자 ↔ 카드 발신자). 쌍 단위로 제한 적용. |
| **언제까지** | “만나기 전” — 매니저가 만남 확정(예: 만남 완료 처리)하기 전까지. 만남 확정 후에는 제한 해제 또는 별도 정책 적용 가능. |
| **몇 회** | **총 4회** 메시지. “4회”는 한 쌍이 주고받은 메시지 수의 합(또는 정책에 따라 “발신 4회” 등으로 정의 가능). 현재 앱 정책: **만나기 전 총 4회 메시지 제한**. |

- 제한은 **클라이언트가 아닌 서버에서 강제**한다. 클라이언트는 표시만 하고, 허용 여부는 API/DB가 결정한다.

---

## 2. 서버 강제 방식 (API/DB 기준)

- **메시지 전송 API**에서만 허용 여부를 판단하지 말고, **메시지 저장 직전(또는 트랜잭션 내)**에서 현재 카운트를 조회한 뒤 4회 초과 시 저장 거부 및 적절한 HTTP 상태/메시지 반환.
- 클라이언트는 “남은 횟수” 등을 **표시용**으로만 사용하고, “전송 가능 여부”의 **진실(truth)**은 서버 응답으로만 판단한다.
- 권장 구조:
  - **메시지 생성/전송 API**: 인증 후, (sender_id, receiver_id) 또는 (match_id / room_id) 기준으로 **현재 메시지 수**를 조회 → 4회 미만이면 insert, 아니면 4xx + `{ message: "..." }` 반환.
  - DB에서 “만나기 전” 구간만 세도록 설계(예: `meeting_confirmed_at IS NULL`인 매칭에 대해서만 카운트).

---

## 3. 데이터 모델에 필요한 필드 예시

서버에서 4회 제한을 적용하려면 아래와 같은 정보가 필요하다. (실제 테이블명/컬럼명은 스키마에 맞게 조정.)

| 목적 | 필드/구조 예시 |
|------|----------------|
| 메시지 수 집계 | `messages`(또는 `chat_messages`) 테이블: `sender_id`, `receiver_id`, `match_id`(또는 `room_id`), `created_at`. 한 쌍의 “만나기 전” 메시지만 카운트할 수 있도록. |
| “만나기 전” 구간 | 매칭/룸 테이블에 `meeting_confirmed_at`(또는 `meeting_done_at`) 같은 필드. NULL이면 “만나기 전”으로 보고, 해당 매칭에 속한 메시지만 4회 제한에 포함. |
| 제한값 상수 | `MAX_MESSAGES_BEFORE_MEETING = 4` 등을 서버(또는 DB 설정)에 두고, API/트리거에서 이 값으로 비교. |

- **선택**: 성능을 위해 “쌍별 현재 메시지 수”를 `match_id`(또는 room) 단위로 캐시/집계 테이블로 유지할 수 있다. 이 경우 **메시지 insert 시 해당 집계를 갱신**하고, 갱신 시 4회 초과면 rollback 또는 insert 거부하도록 설계.

---

## 4. 동시성 / Race condition 대응

- **위험**: 같은 쌍이 동시에 여러 메시지 전송 요청을 보내면, “현재 3회”인 상태에서 두 요청이 동시에 “4회 미만”으로 판단해 둘 다 허용되어 실제로는 5회가 될 수 있음.
- **대응 방법 (택일 또는 조합)**  
  1. **DB 트랜잭션 + 집계 재계산**  
     메시지 insert를 트랜잭션 안에서 수행하고, insert 전에 `SELECT COUNT(*) ... FOR UPDATE`(또는 동등한 잠금)로 해당 쌍의 “만나기 전” 메시지 수를 잠근 뒤, 4회 미만일 때만 insert 후 commit.  
  2. **유니크 제약 + 애플리케이션 재시도**  
     (쌍, 순번) 형태로 유니크 제약을 걸고, insert 시 “현재 최대 순번+1”을 사용. 동시에 같은 순번을 쓰면 한쪽이 실패하므로, 클라이언트는 4xx 시 “제한 초과” 메시지로 처리.  
  3. **쌍 단위 락**  
     Redis 등으로 `match_id`(또는 room_id) 단위 락을 걸고, 메시지 전송 처리 시 락 획득 → 카운트 조회 → 4회 미만이면 insert → 락 해제.

- **권장**: 1번(트랜잭션 + COUNT + 잠금)으로 서버 단일 소스 오브 트루스가 되도록 하고, 클라이언트는 동시에 여러 번 “전송” 버튼을 눌러도 서버가 한 번만 허용되도록 한다.

---

## 5. 클라이언트 UI 표시와 서버 truth 분리

- **서버**  
  - 메시지 전송 API: 4회 초과 시 **반드시 4xx**(예: 429 또는 400)와 `{ message: "만나기 전에는 총 4회까지 메시지를 보낼 수 있어요." }` 등으로 거부.  
  - “남은 횟수”를 반환하는 API가 있다면, **표시용**이며 실제 허용 여부는 “전송 API의 성공/실패”가 기준이다.

- **클라이언트**  
  - 남은 횟수, “4회 중 n회 사용” 등은 **서버에서 받은 값**으로만 표시.  
  - 전송 실패(4xx) 시 서버 `message`를 사용해 토스트/인라인 메시지로 안내.  
  - 클라이언트에서 “4회 넘었으니 버튼 비활성화” 같은 **선제 제한**은 UX용이며, 서버 검증을 대체할 수 없다.

## 6. 테스트 케이스

| 구분 | 시나리오 | 기대 결과 |
|------|----------|-----------|
| **정상** | 한 쌍이 만나기 전에 1~4번째 메시지 전송 | 각 요청 200, 메시지 저장, 응답에 성공 및 (선택) 남은 횟수 반환. |
| **정상** | 4번째 메시지 전송 후 5번째 전송 요청 | 4xx, `{ message: "..." }`, 메시지 미저장. |
| **초과** | 이미 4회인 상태에서 다시 전송 API 호출 | 4xx, 메시지 미저장. (이미 만남 완료된 쌍은 정책에 따라 별도: 만남 후 제한 해제 시 5번째부터 허용 등.) |
| **동시 요청** | 같은 쌍이 거의 동시에 4번째·5번째 메시지 전송 요청 2개 | 서버가 둘 중 하나만 성공(4번째만 저장), 다른 하나는 4xx로 거부. 총 저장 메시지 수는 4를 넘지 않음. |
| **동시 요청** | 같은 쌍이 동시에 3번째 메시지 전송 요청 2개 | 한 건만 성공(3번째 저장), 한 건은 4xx 또는 “이미 제한 도달” 등으로 거부. |

---

## 7. 결제/코인 로직과의 분리

- 채팅 4회 제한은 **결제(PortOne 확인, 코인 충전)** 및 **코인 차감(unlock_profile_card RPC)** 과 **독립**이다.
- 메시지 전송 시 코인을 차감하는 정책이 있다면 별도 문서/정책으로 정의하고, “4회 제한”과 “코인 차감”은 각각 서버에서 독립적으로 검증한다.
